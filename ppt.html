问题 · 01

为什么不能直接让"裸模型"写公文

01

事实幻觉

编造政策文号、编造数据、张冠李戴的部门职责。公文里的"合理编造"是事故，不是瑕疵。

02

格式与文种刚性

GB/T 9704-2012 版头、字体、层次序数，15 种法定文种各行文方向与惯用语——文种用错，文件直接作废。

03

文风偏离

公文要求庄重、简洁、规范，裸模型输出易口语化、冗长拖沓，语气基调与机关行文语体差距明显。

04

长文失控

直接生成长文时，模型缺乏对全文结构的整体把控，容易出现前後矛盾、重复啰嗦、主题漂移。

结论 → 公文写作是一件基于严格的规则的一连串复杂的任务，必须有Agent的能力才能完成。

公文写作在方法论层面，本质是“检索增强的结构化长文生成”——因此我们把研究对象从“公文专属工具”，转向了 Research Agent 这条更成熟的技术脉络。

方法论 · STORM

STORM —— 这条技术脉络的开山之作

Stanford OVAL NAACL 2024 30k+ ★ GitHub

STORM（Synthesis of Topic Outlines through Retrieval and Multi-perspective，即"通过检索与多视角提问的主题大纲合成"）是斯坦福 OVAL 实验室在 NAACL 2024 发表的开源项目，其核心思路是"先检索、后动笔"——通过多视角提问系统性地收集知识，再生成大纲逐节撰写，实现了从零到完整长文的自动生成，已成为 research agent 领域的基础范式。

1

知识整编

检索 + 多视角提问

2

大纲生成

基于整编知识修订

3

文章生成

按大纲逐节撰写

4

文章润色

去冗余、提连贯

STORM 由知识整编、大纲生成、文章生成、文章润色四个模块组成。各模块接口与实现相对独立，可根据具体场景替换或扩展，例如将段落生成调整为要点式输出。

STORM 机制一

视角引导提问

01 · 先找“同类文章”

相似主题文章

不直接检索答案，而是先观察“这类文章通常怎么写”。

DOC同主题文章 A · 工作方案
DOC同主题文章 B · 专项报告
DOC同主题文章 C · 政策解读
02 · 提取结构先验

从目录归纳覆盖维度

TOC（Table of Contents）是前人对“应该覆盖什么”的结构化共识。

01背景与政策依据共现 3/3
02目标与重点任务共现 3/3
03执行分工与保障共现 2/3
04风险与评估机制归纳
03 · 视角驱动提问

把结构变成提问角色

每个视角独立提问，避免单一角度越问越窄。

基础事实 政策依据 执行职责 影响与争议 风险合规

从目录提炼提问角度

目录反复出现“执行分工”→执行职责视角→谁负责？如何分工？权限边界是什么？

公文场景的改造

静态文种级视角库：政策、任务、分工、保障、合规
动态范文补充主题特有视角：如防汛响应分级

好问题不是凭空生成的，而是由“相似文章的结构先验 + 多视角角色 + 检索结果”共同驱动。

附录 · 后续增强方向

Co-STORM：暂不纳入当前主流程

Co-STORM 在 STORM 之上增加多 Agent 协作和用户随时插话能力；当前方案先采用更稳定的多视角检索与证据整编，暂不引入这套机制。

02 · Structural Analysis

段落功能

开头功能

目的说明 + 政策依据 + 发文动作

主体关系

范围 → 内容 → 执行要求

结尾功能

03 · Reusable Skeleton

可复用结构骨架

开头骨架

结尾骨架

收尾 · 下

下一步值得研究的方向

01 / 业务认知

先弄清楚公文到底怎么写

目前的设计仍然停留在想象层面。下一步要从真实公文和真实工作场景出发，弄清楚一份公文为什么要写、需要写哪些内容、由谁参与，以及从起草到审核、签发、流转的完整业务流程。

02 / 框架吸收

持续学习新框架

STORM 是很好的入门，但不应成为终点。持续跟踪 Research Agent、Deep Research、RAG 与协作式写作等新项目和新理念，吸收可迁移的机制。

03 / 交互设计

让用户参与并掌握方向

交互不为炫技，而是让用户能参与取舍、操控方向、核验来源，减少 Agent 自行生成造成的错误，也让最终结果更有把握和信任感。

04 / Agent 协作

设计 Agent 之间的分工

明确知识库、检索、写作、审校和格式渲染 Agent 的边界、交接协议与冲突处理，避免多个 Agent 只是叠加调用。
