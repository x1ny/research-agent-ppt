## 一、问题背景与方法论定位

### 1.1 为什么不能直接让大模型写公文

公文写作看似是一个"长文本生成"任务，但直接调用大模型会暴露两个致命痛点：

**痛点一：事实幻觉。** 大模型在缺乏真实依据的情况下，会编造政策文号、编造数据、张冠李戴地分配部门职责。这在一般写作场景里是"质量问题"，但在公文场景里是"事故"——一份编造了政策依据的公文一旦发出，后果是实质性的行政责任问题，不是可以事后修一修文字的瑕疵。

**痛点二：格式与文种刚性。** 公文有两层硬约束：

- **排版层面**：GB/T 9704-2012《党政机关公文格式》规定了版头、份号、密级、发文字号、标题字体（2 号小标宋）、正文字体（3 号仿宋）、结构层次序数（一、/（一）/1./（1））、成文日期、版记等全部细节，容错空间极小。
- **文种层面**：《党政机关公文处理工作条例》规定了 15 种法定文种（决议、决定、命令、公报、公告、通告、意见、通知、通报、报告、请示、批复、议案、函、纪要），每种文种都有行文方向（上行/下行/平行）的硬性规定和适用情形，以及结构惯例、结尾惯用语的约定俗成。文种用错，文件在程序上就是无效的。

**结论**：不是模型能力不够，而是"裸问裸答"这个交互模式本身不适配公文场景。需要一整套流程，把"事实"和"规矩"都喂给模型，模型只负责最后的成文动作。

**定位管理**：这套系统的合理目标是"AI 出 80 分初稿 + 人工终审"，不是"AI 独立发文"。这个定位可以借用 STORM 团队自己对系统的评价来佐证——他们认为系统产出达不到可直接发表的水准，但对写作前期（pre-writing）阶段的知识整理非常有价值。这个定位同样适用于公文场景，也有助于管理团队内部和上级对系统能力的预期。

### 1.2 问题抽象与类比迁移

调研后发现，市面上没有直接对口的"公文写作 Agent"开源方案。这里有一个关键的思路转折，值得作为叙事主线：**与其纠结"有没有人做过公文 Agent"，不如退一步做一次问题抽象**——识别出公文写作在方法论层面属于哪一类更通用的问题，再去这类问题里找最成熟的方案做类比迁移。

（注：这里最初考虑用"泛化能力"来描述这个动作，但"泛化"在机器学习语境里指模型在未见数据上的表现能力，容易被理解成"模型能力"而不是"我们的方法论选择"，因此改用"问题抽象与类比迁移"这个说法，更准确地表达"主动退一步识别问题本质、再找同构方案迁移"这个决策过程。）

**抽象的结果**：公文写作在方法论层面属于"检索增强的结构化长文生成"问题。支撑这个类比的三条共性：

1. 都需要先检索、整理信息，再动笔——不是纯生成任务；
2. 都强依赖大纲/结构，长文本一次性生成容易失控；
3. 都有幻觉风险，事实类内容需要可溯源。

这类问题在学术界已经被 research agent / deep research 这条技术脉络深入研究，因此我们把研究对象从"公文专属工具"转向了这条更成熟的技术路线，通过类比迁移把方法论落到公文场景。

**总纲一句话**：公文写作不是一个生成问题，而是一个"检索 + 规划 + 事实管理 + 确定性渲染"的系统工程问题，LLM 只是其中的生成器件。

## 二、方法论根基：STORM 技术脉络

### 2.1 长文生成方法谱系（背景铺垫）

在深入 STORM 之前，先明确我们为什么选择"大纲先行、检索驱动"这条路线，而不是其他长文生成范式。调研覆盖的路线包括：

- **Plan-and-Write 范式**：最早由 Yao et al. 提出（故事生成场景），核心结论是"先规划再写"在长文一致性上显著优于端到端生成。这是"大纲先行"这一大类方法的理论起点。
- **递归 / 滚动生成**：RecurrentGPT、Re³（Recursive Reprompting and Revision）等工作，用"摘要 + 记忆"滚动地写超长文本，每写一段就更新全局状态。公文篇幅通常不需要这种超长滚动记忆机制，但其中"每段写完更新已写摘要供下一段参考"的技巧，被我们直接吸收进了分段撰写环节（见 3.6 节的"前文摘要"设计）。
- **AgentWrite / LongWriter**（清华、智谱）：思路是先把任务分解成带字数预算的分段计划，再逐段生成，证明了普通模型通过任务分解就能写出万字以上的连贯长文。其"每段带字数预算"的设计对公文场景有直接参考价值——公文对篇幅比例有隐性规范（例如"情况说明"不应比"工作安排"篇幅更长）。
- **DOC（Detailed Outline Control）**：强调大纲不应只是标题列表，而应是带有"每节要点 + 约束"的详细控制结构。这直接支撑了我们"大纲阶段就要标注每段所需事实"这个设计决策，不是事后追加的工程补丁，而是有方法论依据的设计。

**结论**：公文场景适合"大纲先行 + 检索驱动 + 分段生成"这条路线，STORM 是这条路线里方法论最完整、影响力最大的开山工作，因此作为我们方法论的主要参照。

### 2.2 STORM 总览

STORM（Synthesis of Topic Outlines through Retrieval and Multi-perspective Question Asking）是斯坦福 OVAL 实验室提出的知识整编系统，发表于 NAACL 2024，GitHub 上有 27k+ star，是后续 Co-STORM、OmniThink 等一系列工作共同的起点。它能基于检索从零写出带引用的维基百科风格长文。

STORM 的流程分四个模块：

1. **知识整编（Knowledge Curation）**：通过检索和多视角提问，收集主题相关的信息；
2. **大纲生成（Outline Generation）**：基于整编的知识生成/修订大纲；
3. **文章生成（Article Generation）**：按大纲逐节撰写；
4. **文章润色（Article Polishing）**：去冗余、提升连贯性。

这个四段式流程和我们最初提出的"建库 → 检索范文 → 学习范文列大纲 → 索取信息 → 分段撰写"高度同构，这也是我们把 STORM 作为方法论主要参照的原因。

**核心洞察（用于分享中的点题）**：STORM 证明了长文生成质量的瓶颈在写作之前的知识整编阶段，而不在写作本身。这也是我们把大量设计精力放在检索和大纲阶段、而不是"如何让模型写得更好"这个问题上的原因。

### 2.3 机制一：视角引导提问（Perspective-Guided Question Asking）

**要解决的问题**：让 LLM 围绕主题直接提问，问出来的问题会又浅又同质化（"X 是什么"、"X 有什么特点"这类表层问题），检索回来的信息自然也停留在表层。

**原始机制（三步）**：

1. **调研相似主题的已有文章**：给定主题，先检索一批相关主题的文章，抽取它们的目录结构（TOC）。注意不是检索主题本身的资料，而是"同类文章长什么样"。
2. **从目录中归纳视角**：让 LLM 阅读这些目录，归纳出写这类主题通常存在哪些关注角度，生成 N 个"视角角色"，每个视角是一段 persona 描述（例如"你是一个专注于经济影响的写作者"）。同时固定保留一个"基础事实视角"作为兜底。
3. **每个视角独立驱动提问**：后续的信息收集由 N 个带不同 persona 的 agent 各自并行提问，避免单一视角连续提问导致的"注意力塌缩"（越问越窄）。

**为什么有效**：提问的质量决定检索的上限，而好问题来自"先验的领域结构知识"。STORM 用"相似文章的目录"作为这个先验知识的廉价来源——目录本身就是前人对"这类主题该覆盖什么"的结构化共识。

**我们的领域化改造**：公文场景比维基百科主题有一个天然优势——**STORM 需要"猜"视角，公文的视角几乎是先验确定的**，因为文种是有限集合（15 种），每种文种的关注维度高度收敛。因此设计为两层：

- **第一层，文种级静态视角库（规则化）**：为每类文种预置视角，不需要每次现场归纳。例如"工作方案"类固定包含：政策依据视角（上位文件是什么）、目标任务视角（要干成什么样）、执行落地视角（谁来干、何时干、怎么分工）、保障措施视角（钱、人、机制）、风险合规视角（红线在哪）。
- **第二层，主题级动态视角补充（借鉴 STORM）**：混合检索拿到同类范文后，让 LLM 对比"静态视角库"和"范文实际覆盖的维度"，发现这个具体主题特有的角度（例如同样是工作方案，"防汛工作方案"会多出"应急响应分级"这个静态库没有的维度）。

这是"静态视角库 + 范文动态补充"的组合设计——比 STORM 纯动态归纳成本更低、更可控，也是可以在分享中体现原创性的一个设计点。

**视角机制在流程中的两个落点**：一是在大纲生成阶段，每个视角检查大纲是否覆盖了自己关心的维度（相当于多视角评审大纲）；二是在信息收集阶段，每个视角贡献自己维度对应的事实槽位（例如政策依据视角贡献"需要上位文件名称和文号"这个槽位）。也就是说，事实槽位机制（见 3.5 节）本质上是"由视角驱动生成的槽位"。

### 2.4 机制二：模拟对话（Simulated Conversation）

**要解决的问题**：视角解决"从哪些角度问"，模拟对话解决"怎么越问越深"。

**原始机制**：为每个视角跑一段多轮对话（论文中最多 4-5 轮），双方都是 LLM 扮演：

- **写作者（Writer）**：带着视角 persona，基于**对话历史**提出下一个问题——因此问题会随理解加深而进化（例如从"防汛响应怎么分级"进化到"Ⅲ级响应下乡镇一级的职责边界是什么"）。
- **专家（Expert）**：不允许凭参数化知识直接回答，必须把写作者的问题分解成若干搜索查询，逐个检索，过滤来源可信度，再基于检索结果合成带引用的回答。

每段对话结束后产出一串（问题 → 回答 → 引用来源）的轨迹，汇总后先让 LLM 起草一版大纲草稿，再用对话轨迹去修订大纲——大纲的每个章节都能追溯到具体的问答和来源。

**为什么有效**：两个设计点值得注意——① 对话历史作为"理解状态"，让追问天然指向知识缺口，这是把"迭代深化"这个人类研究行为显式建模的结果；② 专家的"检索强制"是幻觉治理的架构级手段，把"生成"和"事实供给"拆成两个角色，事实只能从检索通道进入。

**我们的领域化改造**：信息收集环节（3.5 节）的朴素实现是"逐个槽位做一次检索"，模拟对话机制给出了升级版——把槽位填充做成"拟稿人 agent ↔ 知识库专家 agent"的多轮对话：

- 拟稿人 agent 带着大纲和槽位清单提问，且能基于已获得的答案追问（链式取证，例如第一轮问到"本市 2025 年出台过老旧小区改造实施意见"，第二轮追问"该意见对资金分担比例的具体规定"）；
- 知识库专家 agent 严格遵守"检索强制"：问题 → 分解为检索式 → 查私有库 → 基于命中内容作答，且**必须附带出处**（文件名、文号、条款号）——公文场景比维基更需要精确到文号的出处链，因为终稿引用政策必须精确；
- 我们比 STORM 多了一个第三方参与者：**当专家 agent 检索不到时，不是像 STORM 那样直接返回"没找到"，而是升级成对真实用户的追问**。也就是说信息收集是"拟稿人 agent、知识库专家 agent、真实用户"的三方对话，真实用户是兜底事实源。这一点把模拟对话机制和 HITL 设计自然缝合在一起。

**成本提醒（分享中的清醒判断，体现工程成熟度）**：STORM 的完整配置是 N 个视角 × M 轮对话，成本不低。公文场景应大幅裁剪：视角数控制在 3 个左右、对话 2-3 轮；简单文种（如普通通知）可以直接退化为单轮槽位检索。也就是说这套机制要**按文种复杂度弹性启用**（具体判定规则见 3.7 节）。

### 2.5 机制三：Co-STORM 的协作话语（演进）

Co-STORM（对应论文 *Into the Unknown Unknowns: Engaged Human Learning through Participation in Language Model Agent Conversations*，Jiang et al. 2024）把 STORM 的信息收集过程升级为一个"协作话语"，核心是三种角色的轮流发言：

- **多个专家 agent**（带视角，回答或追问）；
- **主持人 agent**：不是专家，专门盯着"检索到了但对话里还没聊到"的信息，主动抛出问题，防止讨论变浅、原地打转；
- **真实用户**：可以旁观，也可以在任意时刻插话把讨论方向拉走——这是"混合主导权（mixed-initiative）"，不是固定轮到用户才能发言。

整个讨论过程会实时生成一张**动态思维导图**：把聊过的信息组织成层级化的概念树（节点 = 概念，边 = 父子关系），每个节点都挂着支撑它的原始信息和引出它的问题；地图通过 insert 和 reorganize 两种操作持续更新。最终报告按这张地图作为大纲、逐节生成。

轮次管理协议：默认专家按顺序轮流发言；如果连续多轮专家发言都只是"补充细节/给出潜在答案"这类意图（没有开辟新方向），系统会让主持人介入，强制打破停滞。

**对我们的启发（分享中的钩子设计）**：我们目前的 HITL 是"被动等用户在关卡确认"，Co-STORM 的主持人角色给出了一个更主动的形态——**系统自己发现"这个维度信息不够、该问了"就主动抛出来，而不是走到固定关卡才问**。这是我们大纲生成环节（3.4 节）里明确写入的一个未来升级方向。此外，"混合主导权"（用户可随时打断）也比我们目前"固定关卡"式的 HITL 更灵活，是值得吸收的交互理念。

## 三、整体设计：五环节详解

总体架构分五个环节，前一个是离线阶段，后四个是在线阶段：

```
① 公文库搭建（离线）
   → ② 混合检索 · 查询改写（在线）
   → ③ 范文解析 + 大纲生成 · HITL（在线）
   → ④ 信息收集 · 事实槽位（在线）
   → ⑤ 分段撰写（在线）
```

### 3.1 环节一：公文库搭建

**输入**：内部沉淀的真实公文。
**流程**：

1. **收集 + 脱敏清洗**：真实公文往往涉密或涉及具体人名/单位名，入库前必须脱敏处理，按公开/内部/涉密分级，涉密内容不入库。
2. **按文种分类打标**：对应 15 种法定文种。
3. **LLM 摘要提取**：这一步是关键分岔点，产出**两类资产**：
   - **内容摘要库**：供语义检索用，回答"这篇公文讲了什么"；
   - **结构骨架库**：按文种归纳出的通用框架，供后续"范文解析"环节用，回答"这类公文通常怎么写"。
4. **向量化入库**：完成检索索引构建。

**关键设计理由**："内容"和"结构"分开存，是后面"拆范文而不是抄范文"（3.3 节）的前提。如果只有一个混合的语义库，检索出来的范文只能提供"这个主题聊了什么"，没法单独提取"这类文种通常怎么组织"这个结构性知识。

**现实瓶颈（诚实记录，供团队决策参考）**：公开渠道（政府官网政策文件库、政府公报）只有下行公文，请示/报告这类上行文几乎不公开，因此语料冷启动高度依赖内部积累，数据治理（分级、清洗、打标）是整个方案最现实的瓶颈之一，详见第七部分。

### 3.2 环节二：混合检索与查询改写

**痛点**：用户输入是口语化的（例如"帮我写个消防检查的通知"），和公文语体差距很大，直接拿去检索命中率低，专有名词（部门简称、政策简称）也容易被语义检索模糊掉。

**方案一：查询改写（HyDE + multi-query）**。这个思路最初是团队自己提出的设想（"生成几篇可能的摘要去做检索"），后确认与学术界的 HyDE（Hypothetical Document Embeddings，2022）方法一致：先让 LLM 生成若干篇"假想的目标公文摘要"，再用这些假想文档去做向量检索，因为它们在语体、用词、结构上都比用户原话更接近真实公文，检索效果显著提升。生成多篇（而不是一篇）相当于 multi-query 扩展，从不同角度覆盖语义空间，检索结果取并集融合，进一步提高召回。

**金句**："用假想的答案，去找真实的答案，比用问题去找答案更准。"

**方案二：混合检索（Hybrid Retrieval）**。BM25 关键词检索 + 向量检索 + RRF（Reciprocal Rank Fusion）融合。公文场景里"XX市住建局""棚户区改造"这类专有名词，关键词检索比语义检索更可靠——纯向量检索容易把"棚改"和"旧改"这类近义但不同的政策概念混在一起。

**辅助技术（备查，非核心讲述内容）**：召回后可加一层 **重排（Rerank）**（如 BGE-reranker），召回 top-50、精排取 top-5，是性价比最高的检索质量提升手段。

### 3.3 环节三（上）：范文解析——拆结构而非抄内容

**核心定位句**："范文是拿来拆的，不是拿来抄的。"

检索到范文后，不是把范文喂给模型说"照着写"，而是让模型从中拆出：

- **结构骨架**：开头惯用语、分几段、每段大意、结尾惯用语；
- **文风特征**：句式、用词的整体倾向。

这一步直接呼应 2.3 节的视角机制：**静态文种视角库 + 从范文动态补充特有视角**，共同构成这一环节的方法论基础。

### 3.4 环节三（下）：大纲生成与用户协同（HITL）

**核心定位句**："大纲是方向盘——方向错了，后面写得越细返工越大，所以方向必须和用户共同确认。"

大纲生成的产出不是一段文字，而是一个**结构化对象**，每一段除了标题和要点，还要**标注这一段所需的事实**（这是下一环节"事实槽位机制"能够成立的前提，必须在这一步完成标注动作）。

**HITL 交互设计（三层）**：

1. **可编辑的结构化大纲**：用户可以增删段落、调整顺序、修改每段的要点描述；
2. **逐项确认所需信息**：大纲每段附带"这段需要哪些事实"的清单，用户在确认大纲的同时就能看到"哪些信息系统已有、哪些需要用户补充"，把追问前置到大纲阶段一次性完成，避免写作过程中反复打断；
3. **确认后才进入写作**：大纲确认是流程的强制关卡，不是可跳过的建议。

**佐证**：Co-STORM 是 STORM 之后的核心改进方向，引入人机协作让人参与知识整编过程——说明 HITL 不是能力不足的妥协，而是这个领域公认的演进方向（见 2.5 节）。

**未来升级方向（钩子，呼应 2.5 节 Co-STORM 主持人机制）**：目前的 HITL 是被动等用户在关卡确认，未来可以让系统像"主持人"一样主动提示"这个维度信息不足，建议补充"，甚至借鉴"混合主导权"，让用户可以在生成过程中随时插话调整方向，而不仅是固定关卡。

### 3.5 环节四：信息收集与事实槽位机制

**核心定位句**："模型没有自由发挥事实的空间。"

大纲每段的"所需信息"就是事实槽位。填充按优先级降级：

```
用户已提供 → 私有知识库检索 → 都没有则主动追问用户
```

绝不允许模型在信息缺失时自己编造。

**升级实现（呼应 2.4 节模拟对话机制）**：知识库检索不是一次性查询，而是"拟稿人 agent ↔ 知识库专家 agent"的追问式检索——查到一部分线索后继续追问细节，形成链式取证。知识库专家 agent 遵守"检索强制"：只能基于检索结果作答，且必须附带出处（文件名、文号、条款号）。当知识库检索不到时，缺口升级为对真实用户的追问，用户是兜底事实源（三方对话：拟稿人 agent、知识库专家 agent、真实用户）。

**槽位状态**（工程实现参考）：每个槽位维护三种状态——已填充（附出处）、检索中、待用户补充。

### 3.6 环节五：分段撰写——写作配方

**核心定位句**："成文阶段模型做的是'按框架组装事实'，不是自由创作。"

每一段生成时不是裸调 LLM，而是拼装四份材料：

1. **本段结构框架**（3.3 节拆出来的骨架和口吻）；
2. **本段事实槽位内容**（3.5 节填充好的、带出处的信息）；
3. **前文摘要**（保证承接和口径一致，呼应 2.1 节 RecurrentGPT / Re³ 的滚动记忆思路）；
4. **全局约束**（文种规范用语、禁用表述）。

逐段生成、逐段可返工，是长文一次性生成容易失控这一问题的直接对策（呼应 2.1 节的方法论选择）。

### 3.7 机制的弹性启用规则

STORM 系列机制（多视角、多轮模拟对话）如果按论文原始配置（N 个视角 × M 轮对话）全量启用，公文场景的成本会偏高。因此设计为**按文种复杂度分档启用**，判定维度包括：

- **文种自身复杂度**：如"工作方案""请示"这类涉及多部门协作、多维度考量的文种，复杂度高；"通知""通告"这类结构固定、内容单一的文种，复杂度低。
- **检索到的范文覆盖度**：如果检索到的范文高度一致（同类主题范文结构趋同），可以简化视角归纳步骤；如果范文之间差异较大，说明这个主题存在多种处理方式，需要更完整的多视角流程来做判断。
- **信息缺口数量**：大纲初稿生成后，如果待填充槽位数量少、且大多能被知识库直接命中，可以退化为单轮检索；如果缺口多、涉及多部门信息交叉，则启用多轮追问式对话。

**弹性规则示例**：简单文种（通知、通告等）→ 单一视角、单轮槽位检索；中等复杂度文种（意见、报告等）→ 2-3 个视角、1-2 轮追问；高复杂度文种（工作方案、请示等，尤其涉及多部门协作或大额资金）→ 3 个左右视角、2-3 轮模拟对话式追问。

这个弹性设计也是回应"这么多轮对话成本多高"这类质疑的直接答案。

## 四、下游衔接：润色与格式渲染

写作 agent 产出的是**初稿**，定位是"保真"（事实不错、结构不偏），语言的凝练、公文味的浓淡，交给两个下游环节负责，这是架构上的职责分离，不是没有想清楚的托辞。

### 4.1 润色 Agent（初步思路）

- **生成-评审循环**（学术上对应 Self-Refine / evaluator-optimizer 模式）：润色不是一遍过，而是"评审 agent 按维度挑问题 → 润色 agent 针对性修改 → 再评审"的循环，维度可拆分为语体规范、句式凝练、用词得体等。
- **公文润色的特殊性——"更好"是有标准的**：公文语体的惯用句式、规范表述是相对收敛的集合，所以润色可以"有据可依"，例如从范文库检索同类表述作为改写参照，而不是让模型凭感觉润色。这一点是公文润色和通用文本润色的本质区别。

### 4.2 格式渲染引擎

**核心原则**：内容生成和格式渲染严格分离，LLM 只负责内容文本，不参与排版。

最终的版头、主送机关、成文日期、印章位置等，交给 **docxtpl**（模板占位符方案，简单场景）或 **python-docx**（编程式精细控制，复杂场景）按 **GB/T 9704-2012** 规则做确定性渲染。

**行业参考**：GitHub 上已有 **gov-doc-formatter** 项目——一个基于 LLM agent 的党政机关公文自动排版工具，采用 multi-agent 架构，包含行文规范性判断的 router agent、文本清洗 agent、结构标记 agent、一致性校验 agent 和编排器。它"LLM 只做结构标记、排版由确定性引擎完成"的分工方式，正好验证了"内容与格式分离"这一原则的正确性，值得作为工程实现的参考。

### 4.3 合规与敏感词校对（三层漏斗）

- **基础层**：pycorrector（开源中文纠错，支持 Kenlm/MacBERT/GPT 多种后端），处理错别字层面的问题；
- **规则层**：政治性表述、领导人职务表述、涉密词汇等高风险内容，用敏感词库 + 规则做初筛；
- **复核层**：LLM 复核，处理规则难以覆盖的语境性问题。

**行业类比**：出版行业的专业校对系统本质上就是超大规则库驱动，说明"规则优先、模型兜底"这条路线在高风险文本校对场景是被验证过的可行方案。

**架构建议**：合规检查不应该只在写作全部完成后做一遍后处理，成本较高；应该在分段撰写阶段就注入部分约束（文种规范用语、禁用绝对化表述等），把合规检查前移，后处理只做兜底复核。

## 五、技术选型备查

> 本部分是技术选型的详细依据，正式分享中不必展开，作为被追问时的"弹药"或团队内部技术评审的参考。

### 5.1 检索技术栈

- **混合检索**：BM25 + 向量检索 + RRF 融合，解决专有名词语义检索不敏感的问题。
- **重排（Rerank）**：召回 top-50 后用 cross-encoder 精排取 top-5，中文可用 BGE-reranker 系列，是性价比最高的检索质量提升手段。
- **查询改写**：HyDE（生成假设性文档做检索）+ multi-query（生成多篇假设摘要覆盖不同语义角度）。
- **层次化 / 结构化 RAG**（进阶，供私有知识库参考）：
  - **RAPTOR**：对文档做递归聚类摘要，构建树状索引，可在不同抽象层级检索——适合政策文件库天然的层级结构（中央 → 省 → 市 → 区）。
  - **GraphRAG**（微软）：抽取实体关系构建知识图谱 + 社区摘要，适合回答"某政策的上位依据是什么"这类关系型问题。
- **自我校正检索**：
  - **Self-RAG**：模型自己判断"需不需要检索、检索结果相不相关、生成内容有没有依据"；
  - **CRAG**（Corrective RAG）：检索质量差时触发降级策略。
  这两者与我们"知识库检索不到则升级为用户追问"的降级链路思路一致，学术上有现成框架可以引用。
- **中文 Embedding 选型**：BGE（智源）、GTE（阿里）、M3E 是主流开源选择，BGE-M3 支持稠密 + 稀疏 + 多向量三种模式，选型可参考 C-MTEB 榜单。
- **Chunking 策略**：公文有天然的结构边界（一、二、三 / （一）（二）/ 1. 2.），按结构切分优于固定长度切分。

### 5.2 Agent 编排模式

参考 Anthropic《Building Effective Agents》对常见模式的归纳，本方案的对应关系：

- **Prompt Chaining（链式）**：大纲 → 槽位 → 分段写作，每步输出是下一步输入，中间可插入程序化校验（gate）。
- **Routing（路由）**：文种识别是一个 router，识别后分发到不同文种的子流程/子 prompt。
- **Orchestrator-Workers**：分段写作时，一个编排者按大纲分派段落任务给 worker（可并行写）再汇总；公文段落间常有依赖（后段引用前段数字），建议"章节内串行、章节间并行"折中。
- **Evaluator-Optimizer（生成-评审循环）**：一致性自查、合规复核、润色 agent 均属于这类模式。
- **Workflow vs Agent 之辩**（分享中可作为金句）：公文写作恰恰**不应该**做成自由 agent，而应该做成强约束 workflow——公文的流程是确定的、合规要求是刚性的，自由 agent 的灵活性在这里是负资产。金句："能用 workflow 就不用 agent。"
- **结构化输出**：大纲、槽位清单等中间产物应该用 JSON Schema 约束（function calling / structured output），而不是让模型输出自由文本再解析，便于工程可靠性，也方便前端渲染成可编辑的大纲界面/槽位填充表单。

### 5.3 中文公文领域知识

- **格式标准**：GB/T 9704-2012《党政机关公文格式》。
- **文种规则**：《党政机关公文处理工作条例》规定的 15 种法定文种及其行文方向、结构惯例。这些规则应该做成规则引擎/校验表，而不是靠 LLM 记忆——这是"符号规则 + 神经生成"混合架构（neuro-symbolic）的典型应用场景。
- **对标项目**：
  - **gov-doc-formatter**：基于 LLM agent 的公文自动排版工具，multi-agent 架构（见 4.2 节）；
  - **Kumon-Writing-Assistant**：基于 InternLM 实现的公文写作助手，代表"领域微调"路线；
  - **LexiLaw**（中文法律大模型）：另一个垂直领域微调的完整参考案例；
  - **"公文笔杆子" prompt**（李继刚，流传较广的社区案例）：纯 prompt 工程路线的上限案例，把公文专家的经历、技能、表达特征结构化写入 system prompt，并用 few-shot 示例。作为"对照组"：纯 prompt 路线没有事实来源，必然产生幻觉，从而反过来论证了 RAG + 事实槽位方案的必要性。

### 5.4 技术路线对比：三条路径的成本收益

1. **Prompt 工程（含 few-shot 范文）**：成本最低，但没有事实来源，幻觉不可控，适合探索但不适合生产。
2. **RAG**：本方案的主路线，能覆盖大部分需求。
3. **SFT / LoRA 微调**：成本高，主要价值在"风格模仿"这一个子任务上可能值得投入（见 7.2 节），不建议作为主路线。

**结论**：公文场景用"RAG + 强 prompt + 事实槽位"能覆盖约 90% 的需求，微调只在风格模仿子任务上可能有增量价值。

## 六、评估体系

没有评估就没有迭代，这部分容易被忽略但最能体现工程成熟度。

### 6.1 RAG 评估

采用 **RAGAS** 框架的四个经典指标：

- **Faithfulness**（忠实度）：生成内容是否忠于检索到的内容——这是公文场景幻觉率的直接代理指标，应作为核心监控指标；
- **Answer Relevancy**（回答相关性）；
- **Context Precision**（检索精确率）；
- **Context Recall**（检索召回率）。

### 6.2 LLM-as-Judge

用强模型按 rubric 给生成公文打分，关键工程点：

- **rubric 要拆维度**：格式规范性、文种正确性、语言得体性、事实准确性分开打分，不要用单一综合分；
- **用 pairwise 比较而不是绝对打分**：LLM 的绝对打分不稳定，pairwise 相对更可靠；
- **防位置偏差**：交换 A/B 顺序打两次，取一致性结果。

**可复用的评估设计**：STORM 论文的评估组合——ROUGE、实体召回率、标题实体召回率等相似度指标，外加 Prometheus 模型做 rubric 打分——可以直接借鉴到公文评估体系设计中。

### 6.3 黄金测试集

从真实公文中抽取 50-100 篇，人工撰写对应的"输入意图"，构成（意图 → 标准公文）测试对，作为回归测试集。每次调整 prompt 或更换模型都跑一遍，这个测试集本身是团队的核心资产。

### 6.4 分层验收

- **格式层**：可 100% 自动化验证，直接用规则比对 GB/T 9704；
- **内容层**：半自动化，做实体/数字与事实槽位来源的比对；
- **文风层**：靠 LLM-as-judge + 人工抽检。

## 七、深水区问题与未来方向

这部分是尚未完全展开、但已经识别清楚的关键问题，适合作为团队下一阶段的研究议程，也适合在分享中作为"open questions"呈现思考深度。

### 7.1 语料冷启动与合规

真实公文的公开渠道（政府官网政策文件库、政府公报）只覆盖下行公文，请示/报告这类上行文几乎不公开，因此语料高度依赖内部积累。内部公文可能涉密，入库前需要完整的数据治理流程：**分级**（公开/内部/涉密不入库）、**清洗**（去除具体人名单位名等敏感信息）、**打标**（按文种和主题）。这是整个方案最现实的瓶颈，值得作为独立的项目风险单独管理。

### 7.2 风格模仿的技术选型

"学习范文风格"有四条可选路线：

1. **Few-shot ICL**：把范文直接塞进 prompt；
2. **风格摘要中间表示（风格卡片）**：先让 LLM 提炼"风格卡片"——句长分布、惯用语、语气特征，再注入生成 prompt；
3. **LoRA 微调**；
4. **可控生成**（通过控制信号引导生成风格）。

**倾向判断**：更倾向"风格卡片"路线——可解释、可编辑、可复用（同一单位的风格卡片沉淀下来就是团队资产），比 few-shot 更省 token，比微调成本低得多。这是一个值得在分享中作为原创观点提出的判断，也可以作为后续预研的具体方向。

### 7.3 事实槽位的信任边界

已确定的降级链路（用户输入 → 知识库 → 追问用户）还有两个未解决的问题：

- **知识库内容本身可能过期**（政策被新政策废止）；
- **多个来源之间可能存在冲突**。

**解决方向**：给知识库文档加"效力元数据"（发文日期、是否现行有效、被何文件废止），检索时按效力过滤。这是法律检索系统的成熟做法，可以借鉴北大法宝、威科先行等平台的"时效性标注"设计思路。

### 7.4 上下文工程（Context Engineering）

分段撰写时每段的 prompt 要装载：文种规范 + 大纲 + 本段槽位事实 + 已写段落摘要 + 风格卡片——容易超出预算且信息互相稀释。需要解决的问题：给每类信息定优先级、做压缩（已写段落用摘要而非全文）、防止"lost in the middle"。这是 2024-2025 年从"prompt engineering"演进到"context engineering"的核心话题，是当前的前沿方向。

### 7.5 交互形态

一次性生成 vs 协作式写作？Co-STORM 给出的答案是协作式（见 2.5 节）。落到产品设计上就是：大纲可视化编辑、槽位表单式填充、段落级重写按钮、修改留痕。这部分是前端主导的价值高地，也是团队在这个项目里可以差异化投入的方向。

### 7.6 新一代对标项目与范式之争

STORM 发表于 2024 年，虽然代码库更新放缓，但它演化出的方法论已经在 2025 年的 Deep Research 赛道全面开花，以下项目是下一阶段值得深入研究的方向：

- **OmniThink**（浙大 + 阿里，EMNLP 2025）：STORM 的直系升级，基于 STORM 和 DSPy 实现。核心解决的问题是朴素 RAG 检索的信息缺乏深度、存在冗余，导致生成内容浅薄重复。方案是模拟人类"慢思考"过程，通过迭代的**扩展（expansion）**和**反思（reflection）**构建一棵**信息树**和一个**概念池**来组织多样且相关的信息，并提出"**知识密度**"指标（有效独特内容与文本总长度之比）——这个指标对公文场景很有参考价值，因为公文恰恰要求高信息密度、忌空话套话。**对我们特别实用的一点**：OmniThink 已支持基于 **RAGFlow** 的离线本地检索，适配政务内网这类无法联网的部署场景，代码可直接参考。

- **WebWeaver**（阿里，2025.09）：挑战"大纲先行"的静态范式。指出现有方法的问题是"规划与证据获取解耦的静态流程"和"一次性生成导致的长上下文失效（lost in the middle、幻觉）"。方案是双 agent：**规划者**在动态循环中交替进行"证据获取"和"大纲优化"，产出一个每节都链接到证据记忆库的、**有来源支撑的大纲**；**写作者**则逐节执行"层次化检索 + 写作"。可以直接吸收的两点：① 大纲不是一次生成就冻结，而是随信息收集持续修订，可以纳入 HITL 环节（系统在槽位填充中发现新维度时，建议用户修订大纲）；② "有来源支撑的大纲"设计和我们"带槽位与出处的大纲"设计高度一致，等于有顶会论文背书。

- **辨析：大纲先行 vs 动态大纲，我们的立场**。2026 年初的一些工作（如 AgentCPM-Report）批评"先冻结大纲再写"的范式依赖"初始信息完备"这个经常不成立的假设，容易遇到"洞察天花板"。但这个批评主要针对开放式研究报告场景。**公文恰恰是"初始信息可以做到基本完备、结构高度收敛"的场景**，所以大纲先行在公文场景依然是正确的选择——这个辨析本身可以在分享中体现"我们了解范式之争，并且能论证为什么这个场景该选哪边"，是有说服力的深度展示。

- **LangChain Open Deep Research**：工程实现的最佳参考。基于 LangGraph，把报告生成划分为报告规划、人类反馈、查询生成、搜索、章节撰写、质量评估、报告编译等节点，内置人类反馈机制。最值得借鉴的是**章节级质量闭环**：每章写完后用评审模型判断内容是否充分、准确、相关，输出 pass/fail，fail 则进入"写作、反思、搜索、重写"的迭代，同时设 `max_search_depth` 参数限制最大迭代次数防止无限循环——这是"分段撰写 + 润色循环"的现成工程模板。

- **DeerFlow**（字节跳动开源）：多 agent 编排的参考案例，角色分工为协调器（Coordinator）、规划器（Planner）、研究团队（研究员 + 程序员）、报告员（Reporter），基于 LangGraph 管理状态和通信，强调人机协作。角色划分可映射：规划器 ≈ 大纲 agent，研究员 ≈ 知识库专家 agent，报告员 ≈ 成文 + 格式渲染。

**吸收原则（重要，避免"整体照搬"的误判）**：以上项目全部面向开放网络研究，我们的场景是封闭知识库 + 刚性格式，因此不应整体照搬，而是**拆机制吸收**——OmniThink 拿它的本地检索改造和反思式信息收集，WebWeaver 拿它的证据挂载大纲，LangChain ODR 拿它的章节质量闭环和 HITL 节点设计，DeerFlow 拿它的角色分工。

## 八、可直接落地的 PPT 大纲（17 页）

> 这是历经多轮调整后最终确定的分享大纲，每页给出"讲什么"和"怎么呈现"，供制作 PPT 时直接使用。整体节奏建议 20-25 分钟，P9-P14 是主体，每页控制在 1.5-2 分钟。

**第一部分：方法论铺垫**

- **P1 标题页**：公文写作 Agent：技术探索与整体设计思路。
- **P2 为什么不能直接让大模型写公文**：两个痛点（事实幻觉、格式与文种刚性），各配一个具体案例。结论：需要流程来"喂事实、给框架"。
- **P3 问题抽象与类比迁移**：讲清楚没有对口开源方案 → 做问题抽象 → 转向 research agent 方向，列 2-3 条支撑类比的共性。
- **P4 STORM 总览**：定位为"这条技术脉络的开山之作"（NAACL 2024、斯坦福 OVAL、高 star 数、衍生出 Co-STORM/OmniThink），讲四模块流程。
- **P5 机制一：视角引导提问**：原始机制 + 我们的改造（静态视角库 + 范文动态补充）。
- **P6 机制二：模拟对话**：原始机制 + 我们的改造（拟稿人 ↔ 知识库专家 ↔ 用户三方对话）。
- **P7 演进：Co-STORM 的协作话语**：三角色机制（专家/主持人/用户）+ 动态思维导图，埋钩子——"主持人这个主动发现缺口的角色，对我们的人机协同设计有直接启发"。

**第二部分：总体设计**

- **P8 我们的总体设计：五个环节全景图**：全场锚点页，横向五段式流程图，用颜色区分"离线建库"和"在线生成"两个阶段。

**第三部分：逐环节拆解**

- **P9 环节一：公文库搭建**：收集脱敏 → 分类打标 → 摘要提取（分叉出内容摘要库/结构骨架库）→ 向量化入库，强调"双资产"设计。
- **P10 环节二：混合检索与查询改写**：用户原话直接检索 vs 生成假设摘要检索（HyDE + multi-query）的对比，金句"用假想的答案去找真实的答案"。
- **P11 环节三（上）：范文解析——拆结构而非抄内容**：范文的"X光透视"式讲解，标注开头惯用语/核心论据段/结尾惯用语，回收视角机制。
- **P12 环节三（下）：大纲生成与用户协同（HITL）**：大纲编辑器 mockup，强调大纲即方向盘、三层 HITL 交互，回收 Co-STORM 主持人机制的启发。
- **P13 环节四：信息收集与事实槽位机制**：槽位状态表（已填充/检索中/待补充），"模型不允许自己编"，回收模拟对话的链式取证。
- **P14 环节五：分段撰写——写作配方**：四份输入材料拼装成段，"按配方组装事实"。

**第四部分：收尾与展望**

- **P15 交接下一棒：润色与格式渲染**：职责分离——润色 agent（生成-评审循环，有据可依）、格式渲染引擎（docxtpl，GB/T 9704，非 LLM）。
- **P16 未来研究方向**：OmniThink（本地化检索改造）、WebWeaver（大纲动态修订）、Co-STORM 主持人机制的进一步应用，定位为"下一步要重点研究的方向"。
- **P17 Q&A**

### 金句 / 记忆点清单（讲述时可重点强调）

- "公文写作不是一个生成问题，而是一个'检索 + 规划 + 事实管理 + 确定性渲染'的系统工程问题，LLM 只是其中的生成器件。"
- "长文质量的瓶颈在写作之前的知识整编，而不在写作本身。"（STORM 核心洞察）
- "用假想的答案，去找真实的答案。"（查询改写）
- "范文是拿来拆的，不是拿来抄的。"（范文解析）
- "大纲是方向盘——方向错了，写得越细返工越大。"（HITL 设计）
- "模型没有自由发挥事实的空间，它做的是'按框架组装事实'。"（事实槽位与分段撰写）
- "能用 workflow 就不用 agent。"（架构选型判断）

### 术语澄清记录

**"泛化能力" → "问题抽象与类比迁移"**：最初用"泛化"描述"从公文专属工具转向 research agent"这个决策，但"泛化"在机器学习语境指模型在未见数据上的表现能力，容易被听众误解为"模型能力"而非"我们的方法论选择"。改用"问题抽象与类比迁移"更准确地表达"主动退一步识别问题本质、找同构问题类比迁移"这个动作。

### 9.1 术语表

| 术语 | 说明 |
|---|---|
| 问题抽象与类比迁移 | 主动退一步识别问题的方法论本质，转向同构的成熟技术领域借鉴方案的决策过程 |
| 事实槽位 | 大纲中每段标注的、需要具体事实支撑的信息占位，按"用户输入→知识库→追问用户"降级填充 |
| 范文解析 | 从检索到的范文中拆出结构骨架和文风特征，而非直接复用原文内容 |
| 视角引导提问 | STORM 提出的机制，通过多个带不同关注角度的虚拟角色驱动信息收集，避免提问同质化 |
| 模拟对话 | STORM 提出的机制，通过多轮"写作者-专家"对话迭代加深信息收集 |
| 协作话语 | Co-STORM 提出的机制，专家/主持人/用户三方轮流参与、混合主导权的信息收集过程 |
| 写作配方 | 分段撰写时拼装的四份输入材料：结构框架、事实槽位内容、前文摘要、全局约束 |
| HITL | Human-in-the-Loop，人机协同，本方案中主要体现为大纲确认关卡和事实追问 |
| 弹性启用 | 按文种复杂度动态调整视角数量和对话轮次，避免统一套用高成本机制 |
